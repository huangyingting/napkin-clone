## Findings

### 1. Public share/present/embed pages prefetch a non-existent `/sign-up` route

- **Severity:** Medium
- **Page/workflow:** Public read-only share, presentation, and embed pages.
- **Reproduction steps:**
  1. Open `/share/e2e-fixture-deck-e2efixtureshare01`.
  2. Open `/present/e2e-fixture-deck-e2efixtureshare01`.
  3. Open `/embed/e2e-fixture-deck-e2efixtureshare01`.
  4. Watch browser console/network.
- **Expected:** Public pages should not produce failed route prefetches; sign-up CTAs should target the existing route.
- **Actual:** Each page logged a 404 resource failure for `/sign-up?_rsc=...`. The app's real signup route is `/signup`.
- **Evidence:** Console/network check returned 404s for `http://localhost:4000/sign-up?_rsc=...`; screenshot: [`s19-share-public.png`](ui-audit-2026-06-27-assets/s19-share-public.png).
- **Suggested fix:** Update public-page CTA hrefs/prefetch targets from `/sign-up` to `/signup`, and add a route/link check covering share, present, and embed pages.

### 2. Login page Google CTA is easy to trigger accidentally from form automation and exits the app

- **Severity:** Medium
- **Page/workflow:** `/login`, mixed Google OAuth and email/password login.
- **Reproduction steps:**
  1. Open `/login` with Google OAuth configured in local env.
  2. Fill email and password.
  3. Trigger the first submit button in the form (`button[type="submit"]`).
- **Expected:** The email/password submit path should be unambiguous; Google OAuth should not be the first generic submit target after filling credentials.
- **Actual:** The browser navigated to Google Accounts instead of submitting the credential login. The Google page identified the app as `napkin`, not TextIQ.
- **Evidence:** [`google-oauth-redirect.png`](ui-audit-2026-06-27-assets/google-oauth-redirect.png).
- **Suggested fix:** Ensure the Google CTA is `type="button"` or outside the credentials form, make the credential submit button the only form submit control, and verify OAuth app branding/environment values.

### 3. Signup validation lacks persistent inline feedback for invalid input

- **Severity:** Low
- **Page/workflow:** `/signup` invalid email + short password.
- **Reproduction steps:**
  1. Open `/signup`.
  2. Enter `not-an-email` and `short`.
  3. Submit.
- **Expected:** Durable inline validation should explain the invalid email and password length requirements.
- **Actual:** The page remained on the form with no persistent inline error in the captured page state. Browser-native validation may have blocked submission, but that feedback is transient and easy to miss.
- **Evidence:** [`s04-signup-validation.png`](ui-audit-2026-06-27-assets/s04-signup-validation.png).
- **Suggested fix:** Add visible inline validation messages for email format and minimum password length before or after submit.

### 4. Mobile authenticated navigation is functional but visually crowded

- **Severity:** Low / polish
- **Page/workflow:** Mobile dashboard navigation drawer.
- **Reproduction steps:**
  1. Open `/app` at `390x844`.
  2. Tap the navigation menu.
- **Expected:** Drawer should clearly separate navigation from page content and use descriptive labels for utility controls.
- **Actual:** Drawer works, but the background document list remains highly visible below/behind it, and the keyboard shortcut action appears as a bare `?` row.
- **Evidence:** [`s07-dashboard-mobile-nav.png`](ui-audit-2026-06-27-assets/s07-dashboard-mobile-nav.png).
- **Suggested fix:** Increase overlay coverage/contrast or lock background scroll while open; label the shortcuts row as `Keyboard shortcuts`.

## Workflow Notes

- **Marketing/public:** Home page renders well on desktop and mobile. Anchor navigation works. Public share/present/embed render the seeded visual content, but the `/sign-up` 404 should be fixed before launch.
- **Authentication:** Email/password invalid login has clear feedback. Forgot password uses a safe generic success response. Google OAuth behavior needs cleanup to prevent accidental external navigation and incorrect app branding.
- **Dashboard/documents:** Search, favorites empty state, sorting/filter controls, onboarding, trash, and seeded cards were reachable. Viewer can see shared document access without owner-only share/edit actions.
- **Editor:** Desktop editor, visual card actions, shortcuts, share dialog, version history, export menu, and mobile slide editor were reachable. Save status stayed stable as `All changes saved`.
- **Workspace:** Owner sees members, role management, invite links, documents, workspace settings, and delete controls. Viewer sees membership/leave controls without owner management.
- **Settings/billing/brand:** Profile, password, connected accounts, data export, danger zone, billing plan, credits, and free-plan brand gating were visible.

## UX And Visual Feedback

- Navigation hierarchy is understandable on desktop; mobile relies heavily on icon-only toolbar buttons, though accessible names are present in the DOM.
- Editor toolbar grouping is strong, but dense on mobile; continuing to prioritize frequently used actions would help.
- Empty states are clear for favorites and trash.
- Billing communicates plan limits and locked features clearly.
- Error handling is mostly user-safe; signup validation should be more explicit and public-page route prefetch failures should be eliminated.
- Accessibility basics looked reasonable from snapshots: key controls have accessible names, and read-only states are textually exposed. Follow-up should include keyboard-only traversal and focus-trap checks for dialogs/drawers.

## Final Assessment

**Overall quality rating:** 7/10 for local seeded flows. Core surfaces are reachable and mostly stable, with a few launch-polish issues and one route-link bug.

**Top 5 fixes:**

1. Replace `/sign-up` links/prefetches with `/signup` on public share/present/embed surfaces.
2. Make email/password login submit unambiguous and verify Google OAuth app branding.
3. Add persistent inline signup validation messages.
4. Improve mobile drawer contrast/background scroll handling and label the `?` shortcut item.
5. Add focused automation for public link health, OAuth CTA behavior, signup validation, and mobile navigation state.

**Production-ready areas observed:** public home, seeded dashboard, editor read/write/read-only states, share dialog, version history, billing, trash, and workspace role visibility.

**Follow-up testing:** real email reset delivery, real OAuth completion, file import/upload/download assertions, keyboard-only dialog/drawer focus trapping, cross-browser visual pass, and plan-gated export behavior with paid entitlements.

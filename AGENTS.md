# Ralph Agent Instructions

## Overview

Ralph is an autonomous AI agent loop that runs GitHub Copilot CLI repeatedly until all PRD items are complete. Each iteration is a fresh instance with clean context.

## Commands

```bash
# Run the flowchart dev server
cd flowchart && npm run dev

# Build the flowchart
cd flowchart && npm run build

# Run Ralph with GitHub Copilot CLI
./ralph.sh [max_iterations]
```

## Key Files

- `scripts/ralph.sh` - The bash loop that spawns fresh GitHub Copilot CLI instances
- `scripts/prompt.md` - Instructions given to each Copilot CLI instance
- `prd.json.example` - Example PRD format
- `flowchart/` - Interactive React Flow diagram explaining how Ralph works

## Flowchart

The `flowchart/` directory contains an interactive visualization built with React Flow. It's designed for presentations - click through to reveal each step with animations.

To run locally:
```bash
cd flowchart
npm install
npm run dev
```

## Patterns

- Each iteration spawns a fresh GitHub Copilot CLI instance with clean context
- Memory persists via git history, `progress.txt`, and `prd.json`
- Stories should be small enough to complete in one context window
- Always update AGENTS.md with discovered patterns for future iterations

### Database (Prisma 7 — SQLite local / Postgres prod)

- **ORM:** Prisma **7.x**. The engine is chosen at runtime by **`DB_PROVIDER`**
  (`sqlite` | `postgres`, default **`sqlite`** when unset): **SQLite is the
  zero-setup default for local dev/test** (no DB service to run — data in
  `prisma/dev.db`), and **PostgreSQL is production**. `DATABASE_URL` lives in
  `.env` (git-ignored; see `.env.example`) and is **optional for SQLite**
  (defaults to `file:./prisma/dev.db` when unset). Zero-setup quickstart:
  `npm install && npm run db:migrate && npm run db:seed && npm run dev` (see
  README). Switch to Postgres by setting `DB_PROVIDER=postgres` and a
  `postgresql://` `DATABASE_URL`.
- **Prisma 7 breaking changes (don't trust older memory):**
  - The schema `datasource` block must **not** contain `url`. The connection URL
    lives in `prisma.config.ts` (`datasource.url`, loaded from `.env` via
    `dotenv/config`) for CLI commands (migrate/seed/studio).
  - The runtime `PrismaClient` needs a **driver adapter**, picked by `DB_PROVIDER`
    in `src/lib/prisma.ts` (US-005): `@prisma/adapter-pg` (`pg`,
    `new PrismaPg({ connectionString })`) for `postgres`; `@prisma/adapter-better-sqlite3`
    (`better-sqlite3`, `new PrismaBetterSqlite3({ url })`, default
    `file:./prisma/dev.db`) for `sqlite` (default). `createPrismaClient()` mirrors
    `prisma.config.ts`'s provider/url logic; the global singleton is unchanged.
    The SQLite class is `PrismaBetterSqlite3` (lowercase `qlite`), and its config
    is `{ url }` (not `connectionString`).
  - **The generated client is provider-specific** — the schema's `activeProvider`
    is baked in at generate time, so the runtime adapter **must match** the
    provider the client was generated with, or you get a `PrismaClientInitializationError`
    ("Driver Adapter … is not compatible with the provider … specified in the
    Prisma schema") during `next build`/runtime. Since `postinstall`/`db:generate`
    honor `DB_PROVIDER`, always **generate and build under the same `DB_PROVIDER`**
    (e.g. `DB_PROVIDER=postgres npx prisma generate && DB_PROVIDER=postgres npm run build`).
    Default (unset) generates the **sqlite** client.
  - The client is generated as **TypeScript** into `src/generated/prisma`
    (generator `prisma-client`, not `prisma-client-js`). It is **git-ignored** and
    regenerated by the `postinstall` script (`prisma generate`), so run
    `npm install` (or `npm run db:generate`) after pulling schema changes.
- **Provider selection via `DB_PROVIDER` (US-003).** `prisma.config.ts` reads
  `DB_PROVIDER` (default `"sqlite"` when unset / not exactly `"postgres"`) and
  switches the **schema file** (`prisma/schema.sqlite.prisma` vs
  `prisma/schema.prisma`) and **migrations path** (`prisma/migrations-sqlite` vs
  `prisma/migrations`). `datasource.url` = `DATABASE_URL`, falling back to
  `file:./prisma/dev.db` for SQLite when unset. `prisma/schema.sqlite.prisma` must
  stay **identical to the canonical `prisma/schema.prisma` except
  `provider = "sqlite"`**; `Json` validates on SQLite in Prisma 7 (it maps to
  TEXT). Run CLI commands with the matching env, e.g.
  `DB_PROVIDER=postgres npx prisma validate`. **Gotcha:** on a fresh clone with no
  `DATABASE_URL`, Postgres Prisma CLI commands still need an explicit
  `postgresql://...` URL; otherwise `prisma.config.ts` only has the SQLite fallback
  and Prisma errors with `P1013`.
- **`schema.sqlite.prisma` is GENERATED — don't hand-edit it (US-004).** It's
  derived from the canonical `prisma/schema.prisma` by
  `scripts/gen-sqlite-schema.mjs` (run `npm run db:schema:sqlite`), which copies
  canonical verbatim and rewrites only the datasource `provider` to `"sqlite"`.
  The script is idempotent (output fully recomputed each run → no diff when
  canonical is unchanged). **Gotcha: after editing `prisma/schema.prisma`, rerun
  `npm run db:schema:sqlite`** (and commit the regenerated file) or the two
  engines drift. The only allowed diff between the two files is the one
  `provider` line.
- **Use the shared client:** import `prisma` from `@/lib/prisma` in app code
  (singleton, reused across hot reloads). Don't instantiate `PrismaClient` ad hoc.
- **Migrations — one history per provider (committed).** Postgres lives in
  `prisma/migrations` (lock `provider = "postgresql"`); SQLite in
  `prisma/migrations-sqlite` (lock `provider = "sqlite"`). The provider-aware
  `prisma.config.ts` picks the path from `DB_PROVIDER`. Create/apply with
  `npm run db:migrate` (`prisma migrate dev`); deploy with `npm run db:deploy`.
  Run each under its provider, e.g. `DB_PROVIDER=postgres npm run db:migrate`.
  **The two histories are independent** — a schema change needs a migration in
  *both* (`DB_PROVIDER=sqlite` and `DB_PROVIDER=postgres` `migrate dev`).
- **Converting an enum column to `String` (US-006):** Prisma's auto-generated
  migration does `DROP COLUMN` + `ADD COLUMN` (loses data) and refuses on a
  required column with rows. Generate with `migrate dev --create-only` and
  hand-edit to a data-preserving cast: `ALTER COLUMN "x" DROP DEFAULT;
  ALTER COLUMN "x" SET DATA TYPE TEXT USING ("x"::text); ALTER COLUMN "x" SET
  DEFAULT '…';` then `DROP TYPE "…"`. See
  `prisma/migrations/*_convert_enums_to_string`.
- **Seeding:** `npm run db:seed` (`prisma db seed`); configured in
  `prisma.config.ts` (`migrations.seed = "tsx prisma/seed.ts"`). **Prisma 7's
  `migrate dev` / `migrate reset` NO LONGER auto-seed**, so `db:reset` is
  `prisma migrate reset --force && prisma db seed` (drop → re-migrate → seed,
  non-interactive). `prisma/seed.ts` is **provider-aware** (mirrors
  `src/lib/prisma.ts`: `PrismaPg` for postgres, `PrismaBetterSqlite3` for
  sqlite) — required so `db:reset` seeds on SQLite. Keep the seed idempotent
  (upsert / find-or-create) so reset+seed is safe on both engines. The seed's
  generated-client provider must match `DB_PROVIDER` (US-005 invariant) — run
  `db:seed`/`db:reset` under the same `DB_PROVIDER` the client was generated for.
  The seed also creates a sample **`Visual` with `Json` `data`** (find-or-create by
  `documentId`, type via `VISUAL_KIND_TO_PRISMA`, payload from
  `src/lib/visual/fixtures`) and reads it back through `safeParseVisual` so a broken
  `Json` round-trip (esp. SQLite's `Json`→TEXT) fails the seed loudly. `seed.ts`
  imports app modules by **relative path** (`../src/lib/...`, `../src/generated/...`),
  not the `@/` alias, since `prisma db seed` runs the file directly.
- **Local `.env` gotcha:** this repo's `.env` may set
  `DATABASE_URL=postgresql://…`. Because `dotenv` does **not** override an
  already-set var, SQLite CLI runs on such a machine must pass
  `DATABASE_URL="file:./prisma/dev.db"` explicitly (e.g.
  `DB_PROVIDER=sqlite DATABASE_URL="file:./prisma/dev.db" npm run db:migrate`).
  A fresh clone using `.env.example` (file URL) needs no override. SQLite db
  files (`prisma/*.db`, `-journal`, `-wal`, `-shm`) are git-ignored.
- **Prettier/ESLint** ignore `src/generated` (generated code) — keep it that way.
- **Portable schema — no Postgres-only `enum`s (US-001/US-002).**
  So the same models can target both SQLite (local) and Postgres (prod), the
  schema uses `String` columns instead of Prisma `enum`s, backed by app-level
  string-literal union types. Don't reintroduce `enum`s, and don't import enums
  from `@/generated/prisma`:
  - Visual type: `VISUAL_TYPES` / `VisualType` in `src/lib/visual/schema.ts`
    (uppercase persisted form for `Visual.type`; `VisualKind` is the lowercase
    renderer form). Maps: `VISUAL_KIND_TO_PRISMA` / `PRISMA_TO_VISUAL_KIND`.
  - Workspace role: `WORKSPACE_ROLES` / `WorkspaceRole` / `isWorkspaceRole` /
    `asWorkspaceRole` in `src/lib/workspace/roles.ts`. Since the DB no longer
    enforces the set, coerce role reads at the boundary via `asWorkspaceRole(...)`
    (least-privilege fallback `VIEWER`).
  - `PrismaClient` (class) and the `Prisma` namespace (e.g. `Prisma.InputJsonValue`)
    are still valid generated imports — only the enum exports are gone.

Local Postgres bootstrap — **optional, Postgres-only** (NOT required for local
dev/test, which defaults to SQLite). Only needed when you actually want to run
against Postgres locally (one-time, for a fresh machine):

```bash
sudo apt-get install -y postgresql postgresql-contrib
sudo pg_ctlcluster 16 main start   # or: sudo service postgresql start
sudo -u postgres psql -c "CREATE ROLE napkin LOGIN PASSWORD 'napkin' CREATEDB;" \
                      -c "CREATE DATABASE napkin OWNER napkin;"
# DATABASE_URL="postgresql://napkin:napkin@localhost:5432/napkin?schema=public"
```

### Authentication (Auth.js / NextAuth v5)

- **Library:** `next-auth@beta` (v5, `5.0.0-beta.x`) — officially supports `next ^16`
  and `react ^19`. Passwords hashed with **`bcryptjs`** (pure JS; ships its own types,
  import the default: `import bcrypt from "bcryptjs"`). Cost factor 12.
- **Config:** `src/auth.ts` calls `NextAuth({...})` and exports
  `{ handlers, auth, signIn, signOut }`. Route handler:
  `src/app/api/auth/[...nextauth]/route.ts` (`export const { GET, POST } = handlers`).
- **Session strategy is `jwt`** — the Credentials provider **cannot** use database
  sessions in Auth.js v5. The `jwt`/`session` callbacks copy `user.id` onto the token
  and then onto `session.user.id`.
- **`AUTH_SECRET` env var is required** (in `.env`, git-ignored; placeholder in
  `.env.example`). `trustHost: true` is set so it works locally / off-Vercel.
- **Type augmentation** lives in `src/types/next-auth.d.ts`:
  - `Session` is augmented via `declare module "next-auth"` (adds `user.id`).
  - `JWT` **must** be augmented via `declare module "@auth/core/jwt"` — augmenting
    `"next-auth/jwt"` does **not** merge (it only re-exports), so `token.id` stays
    `unknown` and typecheck fails.
- **Server-side current user:** import `getCurrentUser()` from `@/lib/session`
  (wraps `auth()`, returns the user or `null`). Use in server components/actions.
- **Client-side session:** `src/app/providers.tsx` wraps the app in
  `<SessionProvider>` (mounted in `layout.tsx`); client components call `useSession()`
  from `next-auth/react`. Note: `useSession` resolves a tick after hydration via
  `GET /api/auth/session`, so in browser tests **poll** for the authenticated text.
- **Forms** are server actions + React 19 `useActionState` (login/signup forms are
  `"use client"`; actions are `"use server"` in `actions.ts`). `signIn("credentials",
  { …, redirectTo })` **throws** `AuthError` on bad credentials (catch it → return a
  **generic** "Invalid email or password." message, never reveal which field) and
  throws a `NEXT_REDIRECT` on success (must be re-thrown — don't swallow it).
- Adding `auth()`/`getCurrentUser()` to the root layout (via `SiteHeader`) makes pages
  **dynamically rendered** (they read cookies) — expected; not an error.
- Browser QA tip: dev-tools inject an extra `[role="alert"]`; scope app-error
  assertions to `form [role="alert"]`.

#### Google SSO (OAuth)

- **Provider:** `Google` from `next-auth/providers/google`, registered in
  `src/auth.ts` **only when** `GOOGLE_CLIENT_ID` **and** `GOOGLE_CLIENT_SECRET` are
  set (spread a conditional array into `providers`). Reading them into local
  `const`s first lets TS narrow them to `string` inside the conditional. The
  callback route is the existing `[...nextauth]` handler:
  `/api/auth/callback/google` — add that exact URL as an authorized redirect URI in
  the Google Cloud console.
- **No adapter:** the Credentials provider forces JWT sessions, so there is no
  Account/Session table. Persist the OAuth user **manually** in the `jwt()` callback
  with `prisma.user.upsert({ where: { email } })` (gated on
  `account?.provider === "google"`), then set `token.id = dbUser.id` (the OAuth
  `user.id` is Google's `sub`, **not** our DB id). Linking by email is safe because
  Google emails are verified; it also links a Google login to a pre-existing
  password account with the same email (and keeps the existing `passwordHash`).
  A `signIn()` callback rejects Google sign-ins that arrive without an email.
- **Env vars are `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`** (PRD-specified names,
  passed explicitly as `clientId`/`clientSecret`) — not Auth.js's auto-detected
  `AUTH_GOOGLE_*`. Placeholders live in `.env.example`; local dev uses well-formed
  placeholders in `.env` so the provider registers and the redirect is well-formed.
- **UI:** `src/components/google-sign-in-button.tsx` is a **server** component (no
  `"use client"`) using an inline `"use server"` action `signIn("google", {
  redirectTo: "/" })`; it also exports `OrDivider`. It is rendered by the
  login/signup **page** server components (alongside the client `*-form`), not from
  inside the client form.
- **Browser QA without real Google creds:** click "Continue with Google", intercept
  `page.route("https://accounts.google.com/**", r => r.abort())`, and assert the
  request URL carries our `client_id` + `redirect_uri=.../api/auth/callback/google`.
  Note: dev-browser's QuickJS sandbox `URL` has **no `searchParams`** — parse query
  params with a regex on the raw URL string. Full E2E login needs real credentials.

#### Route protection (Next.js `proxy` + server guards)

- **Next.js 16 renamed `middleware` → `proxy`.** The file is **`src/proxy.ts`** (same
  level as `app/`). Keeping `middleware.ts` builds with a deprecation warning; an
  invalid export there **fails the build**. `proxy` runs on the **Node.js** runtime
  (the `edge` runtime is _not_ supported in `proxy`) — so it can use heavier deps, but
  we still keep it provider-free (see below).
- **Export shape matters (static analysis).** The file must export a function as
  `export function proxy`, `export const proxy = …` (a plain identifier), a
  **default** export, or `export { x as proxy }`. A **destructured**
  `export const { auth: proxy } = NextAuth(...)` is **NOT** detected → build error
  "must export a function". We use:
  `const { auth } = NextAuth(authConfig); export const proxy = auth;` plus
  `export const config = { matcher: [...] }`.
- **Config split:** `src/auth.config.ts` holds the lightweight, provider-free config
  (`providers: []`, `pages`, `session`, `trustHost`, and the `authorized` callback),
  typed with `satisfies NextAuthConfig`. `src/auth.ts` spreads it
  (`...authConfig`, `callbacks: { ...authConfig.callbacks, signIn, jwt, session }`).
  `src/proxy.ts` builds its own `NextAuth(authConfig)` so the proxy never bundles
  Prisma/bcrypt.
- **`authorized({ auth, request: { nextUrl } })`** returns: `false` for unauthenticated
  `/app/*` (Auth.js then redirects to `pages.signIn` = `/login?callbackUrl=…`);
  `Response.redirect(new URL("/app", nextUrl))` to bounce **logged-in** users off
  `/login` and `/signup`; `true` otherwise. `matcher` excludes
  `api|_next/static|_next/image|favicon.ico` (so the auth API/static assets are
  untouched) but still runs on `/`, `/app`, `/login`, `/signup`.
- **Proxy is optimistic — also guard on the server (defense in depth).** Protected
  pages call **`requireUser()`** from `@/lib/session` (returns the user or
  `redirect("/login")`); `/login` and `/signup` page server components call
  `getCurrentUser()` and `redirect("/app")` when already authed. The protected app
  area lives under **`/app/*`**; the placeholder dashboard is `src/app/app/page.tsx`.
- **Logout** uses `signOut({ redirectTo: "/" })` (in `SignOutButton`) → clears the
  session cookie and returns to the marketing home.
- **Browser QA gotcha:** `/login` & `/signup` render the **Google**
  "Continue with Google" `button[type="submit"]` _above_ the credentials form, so a
  bare `button[type="submit"]` selector clicks **Google's** button (its inline action
  POSTs to the page route). Target the credentials submit by text instead
  (`button:has-text("Create account")` / `button:has-text("Log in")`).

#### Documents & dashboard

- **Dashboard** is the protected `/app` route (`src/app/app/page.tsx`, server
  component). The **document editor** lives at **`/app/documents/[id]`**
  (`src/app/app/documents/[id]/page.tsx`) — under `/app/*` so the proxy + a
  server-side `requireUser()` both guard it. The editor page is a thin server
  component that loads the owner-scoped doc and renders the client
  `DocumentEditor` (US-008).
- **Owner-scope every Document query** to prevent cross-user leakage: lists use
  `prisma.document.findMany({ where: { ownerId: user.id } })`; detail/editor pages
  use `findFirst({ where: { id, ownerId: user.id } })` then `notFound()` when null
  (so another user's id 404s instead of leaking). `user.id` comes from
  `requireUser()` (session `user.id`).
- **Create-and-redirect server action** pattern (`src/app/app/actions.ts`,
  `createDocument`): `"use server"` → `requireUser()` → `prisma.document.create`
  → `revalidatePath("/app")` → `redirect(\`/app/documents/${id}\`)`. `redirect`
  **throws** `NEXT_REDIRECT`, so keep it outside any try/catch and call it last.
- **Pending UI for server-action buttons:** wrap the action in a `"use client"`
  component and read `useFormStatus().pending` inside a child of the `<form>` (see
  `src/app/app/new-document-button.tsx`). `useFormStatus` only works in a descendant
  of the form, not the component that renders the `<form>`.
- **Next 16 dynamic route params are async:** type as
  `{ params: Promise<{ id: string }> }` and `await params` (the bundled docs under
  `node_modules/next/dist/docs/` use this shape).
- **Browser QA gotcha:** the dashboard renders a header **"New document"** button
  _and_ (when empty) a **"Create your first document"** button — both submit the
  same action. Target them by their exact text. dev-browser pages are full
  Playwright `Page` objects (`goto`/`fill`/`click`/`waitForURL`/`locator`); a
  `goto()` response's `.status()` is the easy way to assert the 404 scoping case.

#### Document editor — text panel (US-008)

- **Content is stored as Markdown** in the existing `Document.content` String
  field — no schema change, backward compatible with seeded plain text, and
  AI-friendly for later generation. The editor supports only a small block set:
  headings (levels 1–3 = `#`/`##`/`###`), bullet lists (`-`/`*`/`+`), and
  paragraphs.
- **Parsing/rendering lives in `src/lib/markdown.ts`** (`parseMarkdown` →
  typed `MarkdownBlock[]`; `applyBlockType(value, selStart, selEnd, type)` for the
  toolbar). Blocks render via plain React elements in `markdown-preview.tsx`
  (`MarkdownPreview`, directive-free so it's reusable in server views) — **never
  `dangerouslySetInnerHTML`**, so there's no HTML-injection surface. Reuse these
  for the read-only public page (US-017) instead of adding a Markdown dep.
- **`parseMarkdown()` now assigns stable `block.id`s** from each block's normalized
  signature plus per-signature occurrence count. When wiring inline UI/visual
  anchors, key off `block.id` instead of array indexes so unchanged blocks keep the
  same identity across re-parses while edited blocks get new ids.
- **Autosave = debounced (800ms) server actions**, not a form submit:
  `src/app/app/documents/[id]/actions.ts` exports `saveDocumentTitle` /
  `saveDocumentContent`, both `"use server"`, `requireUser()` + owner-scoped
  `prisma.document.updateMany({ where: { id, ownerId }, data })`. Use `updateMany`
  (not `update`) so a foreign id is a harmless no-op rather than a throw/leak.
  `saveDocumentTitle` returns the normalized title (empty → `"Untitled"`).
- **Client editor** `document-editor.tsx` (`"use client"`) owns all interactive
  chrome (inline title `<input>`, Markdown `<textarea>`, block toolbar,
  Write/Preview tabs, save-status `role="status"`). A small `useAutosave` hook
  tracks last-saved value, debounces, exposes `flush` for save-on-blur, and
  reports `saved`/`pending`/`saving`. Toolbar buttons use
  `onMouseDown={(e)=>e.preventDefault()}` to keep the textarea focused/selected;
  after a toolbar edit, restore the caret with a `useEffect` + a
  `pendingSelection` ref (`textarea.setSelectionRange`). Don't reflect saved
  `content`/`title` props back into state after mount — `useState(initial)` is
  seeded once; the post-action router refresh won't clobber local edits.
- **Browser QA:** the editor exposes `aria-label="Document title"` /
  `aria-label="Document text"` and toolbar `aria-label`s (`Heading 1`,
  `Bullet list`, …) — target those. Wait for the `role="status"` text
  `"All changes saved"` before reloading to assert persistence; reload with
  `page.reload()` (hard nav) to bypass the client Router Cache. Heads-up: a
  **stale JWT session cookie** in the named dev-browser can point at a since-wiped
  user → `createDocument` fails with `Document_ownerId_fkey`; `clearCookies()` +
  sign up a fresh user to get a valid `user.id`.

#### Visual data model & renderer (US-009)

- **Canonical visual format = `nodes + edges + style`, versioned JSON.** The
  types, defaults, and validators live in `src/lib/visual/schema.ts`
  (`Visual`, `VisualNode`, `VisualEdge`, `VisualStyle`, `VISUAL_SCHEMA_VERSION`).
  The five `VisualKind`s are lowercase (`flowchart`/`mindmap`/`list`/`chart`/
  `concept`); map to/from the persisted `VisualType` (the app-level union in the
  same module — formerly a Prisma enum, now a `String` column) with
  `VISUAL_KIND_TO_PRISMA` / `PRISMA_TO_VISUAL_KIND`. Persist a visual's JSON into
  the existing `Visual.data Json` column (US-011) — no schema change.
- **Node coordinates are the node _center_** (`x`/`y`), optional. Positioned
  types (flowchart, mindmap, concept) use them; `chart` lays out bars from

#### AI generation endpoint (US-010)

- **`POST /api/generate`** (`src/app/api/generate/route.ts`, `runtime = "nodejs"`)
  turns `{ text, type? }` into `{ candidates: Visual[] }` (3+ visuals validated by
  `validateVisual`). The handler order is **validate → config → quota → generate**:
  parse body → require non-empty `text` → reject `text.length > 10_000` (**413, before
  any LLM call**) → optional `type` must be a `VisualKind` → require `AUTH_SECRET` →
  resolve Azure config (missing → **503**, so misconfig never burns quota) → quota/rate
  limit → `generateVisuals` → 200 (+ set anon cookie). Error map: bad body 400, oversize
  413, quota/rate 429, generation failure 502, misconfig 503/500.
- **Generation core is network-free and injectable.** `src/lib/ai/generate.ts`
  `generateVisuals(input, { complete, maxAttempts?, minCandidates? })` takes a
  `complete(messages) => Promise<string>` so it unit-tests deterministically (the route
  injects the real Azure client). It builds messages, **`extractJson`** (tolerates code
  fences / surrounding prose), **`coerceCandidates`** (`visuals`/`candidates`/`options`/
  `results` array, bare array, or single object), validates each via `safeParseVisual`,
  needs `>= MIN_CANDIDATES` (3), and **retries** up to `maxAttempts` (default 2) before
  throwing `GenerationError`. `MAX_INPUT_CHARS`/`InputTooLongError`/`EmptyInputError` are
  checked **before** calling `complete`. Returns up to `MAX_CANDIDATES` (6), `type`
  matches first.
- **Azure client** `src/lib/ai/azure.ts`: `getAzureConfig()` reads
  `AZURE_OPENAI_ENDPOINT`/`_API_KEY`/`_DEPLOYMENT` (default `gpt-5.5`)/`_API_VERSION`
  (default stable GA), throwing `AzureConfigError` when endpoint/key are missing;
  `azureChatComplete()` POSTs to
  `{endpoint}/openai/deployments/{deployment}/chat/completions?api-version=…` with
  header `api-key` and `response_format: { type: "json_object" }` + `max_completion_tokens`
  (newer models reject `max_tokens`/custom temperature, so we omit them). The prompt
  (`src/lib/ai/prompt.ts`) embeds the schema (derived from `VISUAL_KINDS`/`NODE_SHAPES`)
  and asks for `{ "visuals": [...] }`; it includes the word "JSON" (required by
  json_object mode).
- **Quota = two mechanisms** (`src/lib/ai/quota.ts`, pure + testable). Anonymous: a
  **signed (HMAC-SHA256 over `AUTH_SECRET`) cookie** `napkin_anon` storing `{ id, count }`
  — a **non-resetting lifetime** trial (count lives in the cookie, no datastore, no time
  window; tracked by cookie id, never IP). `signAnonState`/`parseAnonCookie` (returns
  `null` on tamper/wrong-secret/malformed → mint `newAnonState`); increment **only on
  success**, re-sign, set `HttpOnly`+`SameSite=lax` cookie. Authenticated: in-memory
  fixed-window `checkRateLimit(store, userId, { limit, windowMs, now })` keyed by user id
  (module-level `Map` in the route; per-instance, fine for this scope).
- **Tests run via Node's built-in runner + `tsx`** (no new deps): `npm test` =
  `node --import tsx --test "src/**/*.test.ts"`. `tsx` resolves the `@/` tsconfig alias
  and erases `import type` (so tests don't need the generated Prisma client at runtime).
  Put `*.test.ts` next to the module. Cover pure logic with injected fakes — don't hit
  the network. `src/lib/ai/{generate,quota}.test.ts` are the examples.

#### Edit visual text & elements (US-013)

- **Keep `VisualRenderer` directive-free; add interactivity as an overlay.** The editor
  `src/app/app/documents/[id]/visual-editor.tsx` (`"use client"`, `VisualEditor`) renders

- **Inline edit = a `<foreignObject>` HTML `<input>`** placed at the node box (auto-aligns

- **Browser QA:** hotspots are `[data-node-id="<id>"]` with `aria-label="Edit <label>"`; the

#### Real-time collaborative editing (US-019)

- **Self-hosted Yjs websocket server** lives at `scripts/collab-server.mjs` (run
  with `npm run collab`; `COLLAB_PORT`, default 1234). y-websocket v3 **dropped its
  bundled server**, and the official `@y/websocket-server` pulls in `yjs@14`
  (a version skew with our `yjs@13` client), so the server is **hand-written**
  against the **same** libs the browser client uses — `yjs`, `y-protocols`
  (`sync`/`awareness`), `lib0` (`encoding`/`decoding`/`map`), `ws` — guaranteeing
  protocol compatibility. It's the canonical y-websocket protocol (one in-memory
  `Y.Doc` per room = document id; `messageSync`/`messageAwareness`/
  `messageQueryAwareness`; ping/pong liveness). It is **runtime tooling, not part
  of the Next bundle**; do NOT add `// @ts-nocheck` (ESLint bans it — it's a `.mjs`
  so TS isn't checking it anyway). Smoke-test it headlessly: **Node 22 has a global
  `WebSocket`**, so a tiny script with two `WebsocketProvider`s validates sync /
  CRDT merge / presence / reconnect without a browser.
- **Client wiring is `src/lib/collab/`**: `use-collaboration.ts` (`useCollaboration`
  hook + `useYText` text binding + `useDebouncedSave` + `combineSaveStatus`) and
  `y-text.ts` (pure helpers: `applyTextDiff`, `adjustIndex`, `colorFromId`,
  `initialsOf`). `useCollaboration({ room, userName })` creates the `Y.Doc` eagerly
  (`useState(() => new Y.Doc())`, SSR-safe) and the `WebsocketProvider` in an effect
  (client-only; URL from `NEXT_PUBLIC_COLLAB_WS_URL`, default `ws://localhost:1234`).
  It exposes `ycontent`/`ytitle` (`Y.Text`), `ystate` (`Y.Map`, key `visual` = JSON
  string), `status`, `synced`, `ready`, `peers`, `localOrigin`, and `seed()`.
- **React 19 hook lint rules are strict** (`react-hooks/set-state-in-effect`,
  `react-hooks/refs`): you may **not** call `setState` synchronously in an effect
  body, nor assign `ref.current = x` during render. Patterns used here: keep the
  provider in a **ref** (the UI never consumes it — only `status`/`peers`/`ready`
  via state set from *event callbacks*, which is allowed); rely on
  `awareness.setLocalStateField(...)` firing the `change` listener instead of a
  synchronous `setPeers`; assign "latest callback" refs inside a `useEffect`, not
  during render. Subscribe to Yjs in an effect-registered `observe`, never with a
  synchronous `setState` in the effect body.
- **Seeding (DB → room) is once-only and gated.** On `ready`, the editor calls
  `seed({content,title,visual})`, which (under a `meta.seeded` flag + per-field
  empty checks, origin `SEED_ORIGIN`) loads DB values into the shared types only if
  the room is empty. **Editing is disabled until `ready`** (`editable = canEdit &&
  ready`) so no one types before the room is seeded; a **2.5s degraded fallback**
  flips `ready` true even if the server is unreachable (graceful local-only mode).
  Before `ready`, `useYText` shows the DB `initial`; after, the shared `Y.Text` is
  the single source of truth (so "delete everything" stays empty).
- **Text binding = minimal-diff + transaction origins.** `applyTextDiff` writes the
  smallest prefix/suffix-trimmed change so concurrent edits in different regions
  **merge** (a full delete+insert would clobber). Local edits use `LOCAL_ORIGIN`;
  the `observe` handler classifies `transaction.origin === localOrigin` → fire
  `onLocalChange` (debounced DB save via the existing owner-scoped
  `saveDocumentContent`/`saveDocumentTitle`), else it's remote → **re-map the caret**
  through `event.delta` (`adjustIndex`) and restore it in a `useLayoutEffect` so a
  collaborator's edit doesn't jump your cursor. Only **local** changes persist (the
  originator saves; avoids redundant writes / non-owner `updateMany` no-ops).
- **Visual sync rides the same channel.** `VisualPanel` takes `visualMap` (`ystate`)
  + `localOrigin`; `pushVisual` writes `JSON.stringify(visual)` under key `visual`
  (origin `localOrigin`) on every local select/edit, and an `observe` mirrors
  **remote** writes into `setSelected` (own writes are skipped by origin). Per-doc
  it's still one visual; LWW on the map key is fine (the AC's CRDT requirement is
  about *text*). DB persistence stays the existing debounced `attachVisual`.
- **Presence** = `provider.awareness`. Each connection sets
  `awareness.setLocalStateField("user", { name, color: colorFromId(clientID) })`;
  `computePeers` reads `awareness.getStates()` (self first). `presence.tsx` renders
  an avatar stack (`[aria-label="People here"] > span`, initials, self ringed) + a
  connection pill (`Live`/`Connecting…`/`Offline`).
- **Browser QA gotcha — y-websocket also syncs same-origin tabs via
  `BroadcastChannel`.** Two tabs in **one** browser stay in sync (and even merge
  offline edits) *without the server*, so to prove **server-based** collaboration use
  **two separate browser instances** (dev-browser `--browser <name>`) which share no
  BroadcastChannel and can only talk through the WS server. Verified end-to-end:
  cross-browser text + visual sync, concurrent CRDT merge, presence avatar counts,
  and server stop→`Connecting…`→restart→`Live` with offline edits re-synced to a
  fresh client (no data loss). Edits persist to Postgres via the existing autosave
  (durability). Wait for the textarea to be **enabled** (`:not([disabled])`) as the
  "ready" signal before driving edits.

### Render icons inside nodes (parity-gaps US-003)

- **Name → component resolution lives in `src/components/visual/icon-registry.ts`**

### Soft-delete with undo (US-003)

- **`Document.deletedAt DateTime?` is the soft-delete tombstone** (nullable, indexed,

- **EVERY document read excludes soft-deleted rows** with `deletedAt: null`. The shared
  gate `getAccessibleDocument` (in `@/lib/documents`) adds it, so all its callers
  (attachVisual/detachVisual/comments/delete) inherit the exclusion for free. The
  remaining direct reads each got `deletedAt: null`: dashboard lists (`app/page.tsx`),
  editor detail (`app/documents/[id]/page.tsx`), `share/[shareId]`, `embed/[shareId]`,
  workspace list (`workspaces/[id]/actions.ts`). **When you add a new `prisma.document.find*`,

  an action must scope access but can't use `getAccessibleDocument` because it needs the
  soft-deleted row — e.g. `restoreDocument` does `updateMany({ where: { id, deletedAt:
  { not: null }, OR: documentAccessOr(user.id) }, data: { deletedAt: null } })` (the access
  scope lives in the where-clause → a foreign id is a 0-row no-op, no existence leak).
